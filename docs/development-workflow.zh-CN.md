# 开发工作流程

本文档描述 SublinkPro 的完整开发工作流程，从开始任务到提交更改。

---

## 概述

开发工作流程确保每个代码更改在被视为"完成"之前都经过一致的验证、同步和文档化过程。

## 阶段 1: 任务理解与规划

### 编写代码之前

1. **阅读架构指南**:
   - `AGENTS.md` - 架构概览
   - `docs/development.md` - 开发设置和模式
   - `CONTRIBUTING.md` - 贡献工作流程

2. **理解范围**:
   - 哪些层会受影响？(后端、前端、文档、配置)
   - 入口点是什么？
   - 应该遵循哪些现有模式？

3. **识别跨层影响**:
   - 此更改是否影响 API？→ 前端和 skill-sublinkpro 需要更新
   - 此更改是否影响配置？→ 文档和示例需要更新
   - 此更改是否影响用户行为？→ 功能文档需要更新

4. **规划方案**:
   - 需要修改哪些文件？
   - 需要创建哪些新文件？
   - 需要添加哪些测试？
   - 需要更新哪些文档？

### 需要回答的关键问题

- [ ] 我要解决什么问题？
- [ ] 此更改影响哪些层？
- [ ] 我可以重用哪些现有代码/模式？
- [ ] 架构约束是什么？
- [ ] 哪些文档需要更新？

## 阶段 2: 实现

### 代码质量标准

**遵循现有模式**:
- 匹配周围的代码风格
- 尊重层边界 (不要将业务逻辑放在 `routers/` 中)
- 使用现有的工具和辅助函数
- 遵循命名约定

**添加有意义的注释**:
- 解释**为什么**，而不是**是什么**
- 记录复杂的业务逻辑
- 记录跨层契约
- 记录不明显的算法
- 保持注释与代码更改同步

**为以下内容编写测试**:
- 关键业务逻辑
- API 契约
- 权限检查
- 配置语义
- 数据库迁移
- 协议解析
- 数据转换

### 后端开发

```bash
# 本地运行后端
go mod download
go run main.go
```

**关键原则**:
- 标准 Go 风格
- 正确处理错误
- 使用 GORM 进行数据库操作
- 遵循现有服务模式
- 为业务逻辑添加中文注释

### 前端开发

```bash
# 本地运行前端
cd webs
yarn install
yarn run start
```

**关键原则**:
- React 19 + Vite
- Material UI 组件
- 使用 i18next 进行翻译
- 遵循现有主题模式
- 响应式设计 (桌面 + 移动端)
- 支持浅色 + 深色模式

### 国际化要求

**前端**:
- 为 `zh-CN` 和 `en-US` 添加翻译
- 使用 `useTranslation()` 钩子或 `<Trans>` 组件
- 翻译键应该是语义化的，而不是字面文本

**后端**:
- API 响应: 为 Web UI 显示使用 `i18nKey` + `i18nParams`
- 保留 `msg` 字段以实现向后兼容
- 错误消息: 提供机器可读的代码和 i18n 键

**文档**:
- 同时维护 `.md` (英文) 和 `.zh-CN.md` (中文)
- 保持语言切换链接正常工作
- 内容应语义等价，而不是机器翻译

## 阶段 3: 开发后工作流程 (强制性)

**此阶段是强制性的，必须在完成任何代码更改后自动运行。**

完整详情请见: `.agents/skills/post-dev-workflow/SKILL.md`

### 子阶段 3.1: 代码验证

**后端验证**:
```bash
# 格式化
gofmt -w <changed-files>

# 代码检查
golangci-lint run

# 测试
go test ./...
```

**前端验证**:
```bash
cd webs

# 代码检查 (始终必需)
yarn run lint

# 构建 (当路由/资源/构建受影响时必需)
yarn run build
```

**退出条件**: 所有验证命令无错误通过。

### 子阶段 3.2: 跨层同步

**在以下情况下验证同步**:
- 后端 API 更改 → 更新前端 + skill-sublinkpro/reference/api.md
- 前端行为更改 → 验证后端支持
- 配置更改 → 更新文档 + 示例配置
- 功能添加/更改 → 更新功能文档

**详见**: `.agents/skills/cross-layer-sync/SKILL.md` 获取详细检查清单

**退出条件**: 所有受影响的层已识别并同步。

### 子阶段 3.3: 文档同步

**在以下情况下更新文档**:
- 用户可见行为更改 → 更新功能文档
- API 端点更改 → 更新 skill-sublinkpro/reference/api.md
- 配置选项更改 → 更新 docs/configuration.md
- 部署流程更改 → 更新 docs/installation.md

**双语要求**:
- 同时更新 `.md` (英文) 和 `.zh-CN.md` (中文)
- 保持语言切换链接正常工作
- 验证所有链接和代码示例

**详见**: `.agents/skills/doc-sync-check/SKILL.md` 获取详细检查清单

**退出条件**: 所有相关文档均以两种语言更新。

### 子阶段 3.4: 测试执行

**在以下情况下添加测试**:
- 更改关键业务逻辑
- 更改 API 契约
- 更改权限检查
- 更改配置语义
- 更改数据库迁移
- 更改协议解析
- 修复 bug (添加回归测试)

**测试要求**:
```bash
# 运行相关测试
go test ./services/...
go test ./api/...

# 运行完整套件 (如果时间允许)
go test ./...
```

**退出条件**: 为关键更改添加/更新测试，所有测试通过。

### 子阶段 3.5: 变更总结

**准备提交消息**:

```
<type>(<scope>): <subject>

<body>

<footer>
```

**类型**: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`

**在总结中记录**:
- 更改了什么 (高层次)
- 为什么 (正在解决的问题)
- 如何验证 (运行了哪些命令)
- 同步了哪些层
- 更新了哪些文档
- 任何破坏性更改

**示例**:
```
feat(airports): 支持批量订阅更新

添加了带进度跟踪的批量更新对话框。
用户现在可以选择多个机场并并行更新它们。

验证:
- 后端: gofmt ✅, golangci-lint ✅, go test ./... ✅
- 前端: yarn lint ✅, yarn build ✅

跨层同步:
- 后端: api/clients.go, services/subscription_service.go
- 前端: webs/src/api/airports.js, AirportBatchUpdateDialog.jsx
- 国际化: 添加了 zh-CN 和 en-US 翻译
- 文档: 更新了 skill-sublinkpro/reference/api.md

测试:
- 添加了 TestBatchUpdateAirports
- 添加了服务层批量更新测试
- 所有测试通过 ✅

Closes #789
```

## 阶段 4: 提交与拉取请求

### 提交之前

**检查暂存区**:
```bash
git status
git diff --staged
```

**验证**:
- [ ] 仅暂存了预期文件
- [ ] 没有机密信息 (.env, credentials, tokens)
- [ ] 没有运行时数据 (db/, logs/, out/)
- [ ] 没有 node_modules 或 vendor 目录
- [ ] 没有大型二进制文件

**提交**:
```bash
git add <specific-files>
git commit -m "feat(scope): description"
```

### 创建拉取请求

**目标分支**: 始终以 `dev` 为目标 (不是 `main`)

**PR 描述应包括**:

```markdown
## 概述
简要描述此 PR 的作用

## 更改
- 后端: 列出后端更改
- 前端: 列出前端更改
- 文档: 列出文档更改

## 验证
- [ ] 后端: gofmt ✅, golangci-lint ✅, go test ✅
- [ ] 前端: yarn lint ✅, yarn build ✅
- [ ] 跨层同步已验证
- [ ] 文档已更新 (中英文)
- [ ] 测试已添加/更新

## 测试
如何测试此更改:
1. 逐步说明
2. 预期行为

## 相关 Issue
Closes #123
Fixes #456

## 破坏性更改
列出任何破坏性更改 (或 "无")

## 截图 (如果是 UI 更改)
更改前/后的截图
```

## 阶段 5: 审查与合并

### 自动化 PR 检查

`.github/workflows/pr-checks.yml` 在 PR 打开时自动运行:

- 后端: `golangci-lint run`, `go test ./...`
- 前端: `yarn run lint`, `yarn run build`

**如果检查失败**:
1. 修复问题
2. 推送修复提交
3. 在 PR 上评论 `/recheck` 重新触发检查

### 代码审查

**审查者将检查**:
- 代码质量和正确性
- 测试覆盖率
- 文档完整性
- 跨层同步
- 双语一致性
- 架构合规性

**及时处理反馈**:
- 进行请求的更改
- 回复评论
- 处理反馈后重新请求审查

### 合并

批准并通过检查后:
- 合并到 `dev` 分支
- 删除功能分支 (如需要)
- 监控任何问题

## 快速参考: 常见工作流程

### 添加新的 API 端点

1. **实现**:
   - 后端: 在 `routers/` 中添加路由，在 `api/` 中添加处理器，在 `services/` 中添加逻辑
   - 前端: 在 `webs/src/api/` 中添加 API 函数，在视图中使用
   - 国际化: 为 UI 文本添加翻译

2. **开发后工作流程**:
   - 阶段 1: 运行 gofmt, golangci-lint, go test, yarn lint, yarn build
   - 阶段 2: 验证前端 ↔ 后端同步
   - 阶段 3: 更新 `skill-sublinkpro/reference/api.md`
   - 阶段 4: 添加 API 处理器测试
   - 阶段 5: 准备提交消息

3. **提交**: `feat(api): add new endpoint for X`

### 修复 bug

1. **实现**:
   - 编写一个重现 bug 的测试 (应该失败)
   - 修复 bug
   - 验证测试现在通过

2. **开发后工作流程**:
   - 阶段 1: 运行验证命令
   - 阶段 2: 检查修复是否影响多个层
   - 阶段 3: 如果行为不清楚，更新文档
   - 阶段 4: 验证回归测试通过
   - 阶段 5: 准备提交消息

3. **提交**: `fix(scope): description of bug fix`

### 更新文档

1. **实现**:
   - 更新英文 `.md` 文件
   - 更新中文 `.zh-CN.md` 文件
   - 验证链接和示例

2. **开发后工作流程** (仅文档的简化版):
   - 阶段 1: 跳过 (无代码更改)
   - 阶段 2: 跳过 (仅文档)
   - 阶段 3: 验证双语一致性，测试所有链接
   - 阶段 4: 跳过 (不需要测试)
   - 阶段 5: 准备提交消息

3. **提交**: `docs(scope): description of doc update`

### 添加新协议

1. **实现**:
   - 创建 `node/protocol/myprotocol.go`
   - 实现 Protocol 接口
   - 实现 ProxyCapable, SurgeCapable (如适用)
   - 在 `init()` 中注册
   - 添加单元测试

2. **开发后工作流程**:
   - 阶段 1: 运行 gofmt, golangci-lint, 协议测试
   - 阶段 2: 通常无前端更改 (动态注册)
   - 阶段 3: 如果是面向公众的功能，更新文档
   - 阶段 4: 验证协议测试通过
   - 阶段 5: 准备提交消息

3. **提交**: `feat(protocol): add support for MyProtocol`

## 对于 AI Agent: 自动化说明

### 自动触发

完成代码实现后:

```
IF code_implementation_complete:
    自动调用 .agents/skills/post-dev-workflow/SKILL.md
    
    运行阶段 3.1: 代码验证
    IF 失败:
        修复问题
        重试验证
    
    IF multi_layer_change:
        运行阶段 3.2: 跨层同步
    
    IF behavior_api_config_changed:
        运行阶段 3.3: 文档同步
    
    IF key_logic_changed:
        运行阶段 3.4: 测试执行
    
    运行阶段 3.5: 变更总结
    
    向用户报告结果
    
    IF all_phases_pass:
        声明 "工作已准备好提交"
    ELSE:
        声明 "发现问题，正在修复..."
        修复问题
        重试工作流程
```

### 切勿跳过工作流程

❌ **不要**:
- 在不运行开发后工作流程的情况下声明工作"完成"或"完毕"
- 因为"这是小改动"而跳过验证
- 仅更新英文文档
- 在不检查前端影响的情况下更改后端
- 对"简单"更改跳过测试

✅ **要**:
- 在代码更改后始终运行开发后工作流程
- 在继续之前修复所有验证错误
- 一起更新所有受影响的层
- 维护双语文档
- 为关键逻辑更改添加测试

## 此工作流程的好处

1. **一致性**: 每个更改都经过相同的验证过程
2. **质量**: 在问题到达生产环境之前尽早发现
3. **完整性**: 没有半成品或缺失的文档
4. **可维护性**: 未来的开发者理解更改了什么以及为什么
5. **协作**: 为 AI 和人类贡献者设定明确的期望
6. **效率**: 自动化检查减少手动审查负担

---

## 结论

此工作流程确保 SublinkPro 中的每个代码更改都符合质量标准，尊重架构边界，并保持完整的文档。通过始终如一地遵循此工作流程，我们维护了一个易于理解、维护和贡献的高质量代码库。

**记住**: 开发后工作流程 (阶段 3) 是**强制性的**和**自动的**，在完成任何代码更改后。这不是可选步骤或建议——它是"完成"含义的一部分。
